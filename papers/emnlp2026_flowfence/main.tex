\begin{abstract}
LLM agents increasingly retrieve text from memory or knowledge bases and serialize it into ReAct observations. In agentic settings, retrieved text is not only evidence: once exposed to the model, it can function as executable context. We formulate retrieval-memory poisoning as a \emph{retrieval-to-exposure} boundary problem and introduce \ff, a runtime containment layer inserted after retrieval and before prompt construction. For each retrieved record, \ff assigns a release, rewrite, or quarantine decision, constructs the model-visible view, and records an audit trace linking retrieval, exposure, action, and outcome.

We evaluate \ff on an AgentPoison-derived StrategyQA full-ReAct axis across MiniMax, Qwen, and Kimi model profiles, together with retrieval-pressure slices, paraphrase and adaptive stress tests, lexical and prompt-isolation comparators, inspector-swap replay, benign false-quarantine replay, audit cases, and fixed-trace overhead replay. On Qwen and Kimi, no defense retrieves and exposes poisoned memory at rate 0.96, with attack manifestation rates of 0.693 and 0.893; \ff keeps raw poisoned retrieval observable while reducing exposed poisoned retrieval and manifestation to 0.0. Inspector-swap replay shows that the same boundary can be exercised with rule, heuristic, and LLM-judge inspectors, revealing precision--cost trade-offs. In a Kimi adaptive-poisoning pilot, no defense and a non-oracle static filter expose corrected poisoned content in 0.96 of cases across three harder same-axis families, whereas \ff exposes 0.0. Fixed-trace replay shows about 14 microseconds of local containment work per retrieval event. These results support a focused claim: poisoned retrieval need not disappear for model-visible exposure and downstream manifestation to be suppressed.
\end{abstract}

\section{Introduction}

Tool-using LLM agents routinely retrieve text from memory, databases, search tools, or knowledge bases and paste it into the next prompt or ReAct observation. ReAct-style agents interleave language-model reasoning with actions against external environments \citep{react}, and tool-using language models increasingly rely on external APIs or memories while solving tasks \citep{toolformer}. In standard retrieval-augmented NLP, the central question is often whether retrieved evidence is relevant \citep{lewis2020rag,dpr}. In agentic settings, however, retrieved text is not merely evidence. Once serialized into the active prompt, it can steer tool calls, alter the reasoning trajectory, and change the final answer.

This creates a distinct failure mode for retrieval-memory poisoning. Indirect prompt injection work shows that malicious external content can override an application's intended behavior once it enters the model input \citep{greshake2023indirect}. AgentPoison shows that poisoning an agent's memory or knowledge base can manipulate downstream behavior \citep{agentpoison}. AgentDojo and Agent Security Bench further show that prompt injection and tool-mediated attacks are realistic evaluation targets for LLM agents \citep{agentdojo,asb}. These settings motivate an operational distinction that many pipelines blur: a record being retrieved is not the same event as that record being exposed to the model.

We argue that \emph{retrieval is not exposure}. A poisoned record may be legitimately returned by the retriever while still being unsafe to place in the model-visible context. A defense that removes poisoned records before retrieval tests a different property from runtime containment after retrieval. The relevant boundary question is whether unsafe retrieved records cross into the model-visible context where they can influence reasoning and action.

This paper introduces \ff, a runtime containment layer for retrieval-memory poisoning in LLM agents. \ff is placed after retrieval and before prompt construction. For each retrieved record, it inspects the raw text, assigns a release, rewrite, or quarantine decision, constructs a safe view for the model-visible context, and records an audit trace with decision reasons and content hashes. For ReAct agents, \ff also applies action canonicalization so that quarantine does not destabilize the next tool-use step.

\ff focuses on the runtime transition from retrieved text to model-visible context. Rather than replacing poison detection, it makes exposure decisions explicit, auditable, and measurable after retrieval. This lets us evaluate non-vacuous containment: poisoned records can remain retrievable while exposed poisoned retrieval and downstream attack manifestation are suppressed.

We evaluate \ff on an AgentPoison-derived StrategyQA full-ReAct axis that preserves the upstream ReAct loop and DPR-backed retrieval environment while creating stable attack pressure. Across MiniMax, Qwen, and Kimi model profiles, no-defense runs repeatedly retrieve and expose poisoned memory, whereas \ff keeps raw poisoned retrieval non-zero while suppressing model-visible exposure and attack manifestation. We further analyze mechanism ablations, retrieval pressure, inspector swaps on saved retrieval events, lexical and prompt-isolation comparators, paraphrased and adaptive poisoned instructions, benign false-quarantine behavior, audit cases, and fixed-trace overhead replay.

Our contributions are:
\begin{itemize}
    \item We formulate retrieval-memory poisoning as a retrieval-to-exposure boundary problem and propose \ff, a runtime boundary with release, rewrite, quarantine, ReAct action canonicalization, and audit trace support.
    \item We define non-vacuous containment metrics that distinguish raw poisoned retrieval, model-visible exposed poisoned retrieval, and downstream attack manifestation.
    \item We provide a controlled evaluation across three model profiles and complementary stress settings, showing exposure containment under non-zero poisoned retrieval, inspector precision--cost trade-offs, and low local replay overhead.
\end{itemize}

\section{Problem Setup}

\subsection{Agentic Retrieval-Memory Poisoning}

We consider a text-first, tool-using LLM agent that follows a ReAct-style loop. At each step, the model emits a thought and an action such as \texttt{Search[entity]}, \texttt{Lookup[keyword]}, or \texttt{Finish[answer]}. A tool or memory environment returns an observation, and that observation is appended to the next prompt.

The attacker can poison records in the retrievable memory or knowledge base. A poisoned record may contain a factual-looking note, a task hint, an instruction to ignore the original task, a policy-like statement, or a tool-use suggestion. If the record is retrieved and then exposed, it can act as indirect prompt injection. The attacker does not compromise the model weights, the operating system, the evaluator, or the \ff runtime.

The defense operates at runtime. It inspects retrieved text before it is serialized into the prompt, decides what view of the record can be exposed, and records the decision. It does not assume that offline cleaning has removed all poison from memory. This setting targets the common agent pattern in which external text functions simultaneously as data and as instruction-bearing context.

\subsection{Retrieval, Exposure, and Outcomes}

Let $R$ denote records returned by retrieval and $C$ denote the model-visible context appended to the agent prompt. Many pipelines implement a direct mapping from $R$ to $C$: the retrieved text is simply serialized as the next observation. \ff inserts a boundary between these events.

We distinguish three outcomes:
\begin{itemize}
    \item \textbf{Raw poisoned retrieval}: a poisoned record appears in $R$ before defense processing.
    \item \textbf{Exposed poisoned retrieval}: poisoned content appears in $C$ after defense processing.
    \item \textbf{Attack manifestation}: the adversarial behavior appears in the agent's final answer or trajectory.
\end{itemize}

The desired containment pattern is not zero raw poisoned retrieval. Instead, the non-vacuous target is:
\begin{equation}
\text{RawPoison} > 0,\quad \text{ExposedPoison}=0,\quad \text{Manifestation}=0.
\end{equation}
This pattern shows that the attack reached the retrieval layer but did not cross into model-visible context.

This distinction also separates two system questions that are often conflated. A retriever may still be useful even when poisoned records remain in its candidate set; suppressing all suspicious retrieval can hide the failure mode and destroy auditability. A runtime boundary instead asks what safe view, if any, should be exposed for each retrieved record while preserving enough provenance to inspect what was retrieved and why it was withheld.

\subsection{Design Invariants}

\ff is organized around three invariants.
\textbf{First, raw retrieval is preserved as evidence.} A quarantined record is not silently deleted; its hash, source, preview, and decision are retained in the audit trace.
\textbf{Second, model-visible exposure is explicit.} Every retrieved record has a release, rewrite, or quarantine state before prompt construction.
\textbf{Third, containment is evaluated downstream.} A defense is not successful merely because it flags text; it must reduce exposed poisoned retrieval and attack manifestation while keeping utility effects visible.
These invariants distinguish \ff from a pure blocklist or a prompt-only instruction such as ``do not follow retrieved text.'' The boundary is represented in runtime state and logs, not only in natural-language prompting.

\subsection{Exposure Decisions}

For each retrieved record $r$, \ff produces $d(r)=(\mathrm{state},\mathrm{risk},\mathrm{reasons},\mathrm{view})$. The state is one of:
\begin{itemize}
    \item \textsc{release}: expose the record or an equivalent safe view.
    \item \textsc{rewrite}: expose a sanitized safe view while keeping the original record auditable.
    \item \textsc{quarantine}: withhold the original record from the model-visible context.
\end{itemize}

The current implementation uses explicit inspection signals, including trigger-sequence matches, answer-override language, imperative cross-task instructions, and retrieved payloads that embed another task. These signals produce reason codes such as \texttt{trigger\_sequence\_detected}, \texttt{override\_instruction\_detected}, and \texttt{retrieved\_cross\_task\_payload}. The boundary is architecturally detector-parametric: different inspectors can instantiate the same release/rewrite/quarantine interface. Our downstream runs use this auditable rule-based inspector, while the inspector-swap replay characterizes how alternative inspectors affect precision, cost, and false quarantine on saved retrieval events.

\subsection{Quarantine and Safe Context Construction}

When \ff releases a record, its safe view is appended to the ReAct observation. When it rewrites a record, the model sees only the sanitized safe view. When it quarantines a record, the model instead sees a controlled observation such as:
\begin{quote}
\small Retrieved memory was quarantined by \ff. Continue reasoning from the question and any non-quarantined evidence.
\end{quote}
The original unsafe record is not exposed, but its hash, preview, decision, and reason codes remain in the audit trace. This is different from deletion: deletion removes evidence that a poisoned record was retrieved, whereas quarantine preserves the raw retrieval event while preventing model-visible exposure.

\subsection{ReAct Action Canonicalization}

Quarantine can protect the prompt from unsafe memory, but it can also perturb the ReAct trajectory. After an observation is withheld, the model may emit malformed actions, repeat searches, or carry contaminated text into the next action line. \ff therefore includes a rule-based ReAct action canonicalization layer.

The canonicalizer extracts the last valid tool call of the form \texttt{Search[...]}, \texttt{Lookup[...]}, or \texttt{Finish[...]}; normalizes the tool name and argument; and writes back a clean action string to the next prompt. If the previous step was quarantined, the runtime can also rebuild the next search context from the original question and normalized search query rather than from the full accumulated trace. Quarantine is the primary containment mechanism; ReAct action canonicalization is a trajectory-hygiene component that stabilizes post-containment tool use.

\subsection{Audit Trace}

\ff records an audit entry for each inspected retrieval event. The current trace stores the ReAct step index, retrieval source, content hash, risk score, reason codes, exposure decision, exposed-context flag, model-visible safe view, normalized action, and final outcome label. These fields allow us to ask whether a poisoned record was retrieved, whether it became model-visible context, and what downstream action or answer followed.

\section{Experimental Setup}

\subsection{AgentPoison-Derived ReAct Axis}

The main experiments use an AgentPoison-derived StrategyQA full-ReAct setup \citep{agentpoison}. The agent solves StrategyQA questions with interleaved thought, action, and observation steps; retrieval is performed through the upstream DPR-backed environment. Each formal run uses 25 fixed StrategyQA questions. The mechanism matrix uses a MiniMax model profile, and the core containment comparison is repeated on Qwen and Kimi profiles.

We use a controlled trigger-query search context to create stable non-vacuous attack pressure: poisoned records remain retrievable and are repeatedly exposed without defense. This design isolates the retrieval-to-exposure question that \ff targets---whether unsafe retrieved records cross into model-visible context---rather than measuring whether retrieval itself can avoid poisoned records.

\subsection{Conditions and Metrics}

We compare five conditions in the mechanism matrix: no defense, static keyword filtering, safe-view rewrite only, \ff quarantine only, and \ff quarantine plus action canonicalization. The cross-provider table uses no defense, static keyword filtering, and \ff quarantine plus action canonicalization. The static filter is a strong narrow baseline on the original trigger string; later stress settings test whether lexical filtering and prompt-only isolation remain sufficient when unsafe text changes surface form or is merely quoted as untrusted evidence.

We report raw poisoned retrieval (Raw), exposed poisoned retrieval (Exp.), and attack manifestation (Man.). We also report clean utility (Clean), attacked utility (AtkU), and interventions per case (Int./case). Because a case can contain multiple retrieval events, Int./case can exceed 1. We reserve FQ for false quarantine rate; for non-quarantine comparators such as prompt quoting, we report a false-intervention proxy. Clean and attacked utility are exact-match rates. Unless otherwise stated, results are averaged over three runs.

\subsection{Replay and Pilot Evaluations}

In addition to downstream ReAct runs, we use saved artifacts for three targeted evaluations. First, an inspector-swap replay applies rule, heuristic, and LLM-judge inspectors to saved retrieval events. Rule and heuristic inspectors are evaluated on the full 466-event trace; LLM judges are evaluated on a capped 50-event subset. Second, a benign false-quarantine replay applies the rule and heuristic inspectors to 90 benign records across three families. Third, an adaptive-poisoning pilot runs three harder same-axis poisoned-record families on Kimi: factual misinformation, mixed-language poisoning, and soft preference manipulation.

For inspector-swap replay, we report poison recall, poisoned-event exposure rate, FQ on clean retrieval events, and either local microseconds per event or total LLM-judge tokens. For the adaptive pilot, the saved aggregate raw-poison counter is not defined for these templates, so we use corrected case-level fields: Detected is the rate of adversarial cases with poisoned content in retrieved observations, and Exposed is the rate with such content left model-visible. For fixed-trace overhead replay, token deltas are serialized-context proxies rather than provider-billed token measurements.

\section{Related Work}

\paragraph{ReAct and tool-using agents.}
ReAct introduced interleaved reasoning and action generation for language-model agents \citep{react}. Toolformer showed that language models can learn to invoke external tools during generation \citep{toolformer}. \ff targets a consequence of this design: retrieved observations can become executable model context, not just evidence.

\paragraph{Prompt injection, memory poisoning, and agent security benchmarks.}
Indirect prompt injection work showed that malicious instructions in external content can compromise LLM-integrated applications \citep{greshake2023indirect}. AgentPoison studies poisoning of agent memory and knowledge bases \citep{agentpoison}. AgentDojo and Agent Security Bench provide dynamic agent settings for evaluating attacks and defenses \citep{agentdojo,asb}. G-Safeguard studies security and propagation in multi-agent systems \citep{gsafeguard}. \ff focuses on one narrower point in this space: the runtime boundary between retrieval and model-visible exposure.

\paragraph{Retrieval-augmented NLP and agent memory.}
RAG and dense passage retrieval connect language models to external corpora \citep{lewis2020rag,dpr}. Agent memory systems such as A-Mem are designed to improve long-horizon agent behavior through structured memory use \citep{amem}. MEXTRA studies privacy risks in LLM agent memory \citep{mextra}. Our setting treats retrieved memory as a potentially contaminated text source and asks which retrieved records should become model-visible context.

\paragraph{Filtering, provenance, and auditability.}
Simple lexical filtering and prompt-level isolation are common operational responses to unsafe text, and our experiments show that they can be useful but incomplete. \ff differs by representing exposure decisions explicitly and preserving quarantine traces. This connects to provenance work that records how data influences downstream outputs \citep{buneman2001why,cheney2009provenance}. In agentic NLP, the analogous question is whether retrieved text became model-visible and influenced the subsequent trajectory.

\section{Conclusion}

Retrieval-memory poisoning creates a boundary problem for LLM agents: a record can be correctly retrieved yet unsafe to expose as executable context. \ff inserts a runtime containment layer at that retrieval-to-exposure boundary. On the evaluated AgentPoison-derived axis, \ff keeps raw poisoned retrieval observable while suppressing model-visible exposure and attack manifestation across three model profiles. Stress tests show that prompt-level isolation and broader lexical matching can still leave unsafe text model-visible; replay and pilot evaluations characterize inspector trade-offs, benign false-quarantine behavior, adaptive-family exposure containment, and local overhead. The result is a measurable runtime boundary for containment and audit in ReAct-style agents.

\section*{Limitations}

Our claims are scoped to text-first, ReAct-style retrieval-memory poisoning on a controlled AgentPoison-derived axis. \ff separates the boundary mechanism from the inspector used to instantiate it: downstream experiments use an auditable rule-based inspector, and replay results show how alternative inspectors change precision and cost. The adaptive pilot covers three Kimi families and uses exposure as the primary endpoint, while the benign replay is a template-based false-quarantine check rather than a deployment-scale utility study. Fixed-trace replay isolates local containment overhead and context-length deltas rather than end-to-end provider latency. These choices leave broader agent architectures, full-benchmark reproductions, and deployment-scale utility studies to future work.

\section*{Ethical Considerations}

This work studies defenses against poisoned retrieval memory in LLM agents. The experiments use benchmark-style StrategyQA tasks and synthetic poisoned instructions rather than private user data. The attack examples are limited to controlled instruction-following failures such as forcing an incorrect final answer. Releasing artifacts should avoid API keys, non-anonymized local paths, and live service credentials. \ff can reduce model-visible exposure of unsafe retrieved text, but it should be deployed as one part of a broader safety stack in high-stakes systems.
